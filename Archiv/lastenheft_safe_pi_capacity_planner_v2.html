1. Einleitung

1.1 Zweck des Dokuments

Dieses Lastenheft beschreibt die vollständigen Anforderungen an die Neuentwicklung des SAFe PI Capacity Planners als webbasierte Anwendung. Es richtet sich an externe Entwicklungsdienstleister und bildet die verbindliche Grundlage für Angebot, Vertrag und Abnahme.

Das Dokument definiert, was das System leisten muss (funktionale Anforderungen), wie es strukturiert sein soll (Architektur und ADRs), unter welchen Bedingungen es betrieben wird (nicht-funktionale Anforderungen) und welche Abnahmekriterien erfüllt sein müssen. Die Wahl konkreter Implementierungsdetails liegt beim Auftragnehmer, sofern alle Anforderungen eingehalten werden.

1.2 Ausgangslage

Das Bundesamt für Informatik und Telekommunikation (BIT) setzt zur Software-Entwicklung das Scaled Agile Framework (SAFe) ein. Alle sechs bis acht Wochen findet ein PI Planning statt, in dem Teams ihre verfügbare Kapazität in Story Points (SP) berechnen und Aufgaben für das kommende Program Increment (PI) planen.

Bisher erfolgte diese Kapazitätsplanung manuell in Excel. Dies führt zu Medienbrüchen, hohem Pflegeaufwand und fehlender Transparenz für alle Beteiligten. Eine webbasierte Lösung soll den Prozess digitalisieren, standardisieren und für mehrere Teams gleichzeitig zugänglich machen.

Eine erste Version des SAFe PI Capacity Planners wurde als Proof of Concept realisiert (React/Express/TypeScript, Deployment auf Vercel + Railway, Features 01–23 implementiert). Dieses Lastenheft beschreibt die Anforderungen für eine produktionsreife Neuentwicklung auf Basis der gewonnenen Erkenntnisse.

2. Projektziel und Systemübersicht

2.1 Projektziel

Der SAFe PI Capacity Planner ist eine webbasierte Mehrbenutzerlösung zur Kapazitätsplanung für SAFe PI Planning in der Bundesverwaltung. Das System ermöglicht es Teams, ihre verfügbaren Story Points pro PI und Iteration zu berechnen, indem es individuelle Mitarbeiterabsenzen, Feiertage, Schulferien, Betriebsaufgaben und Pikett-Verpflichtungen berücksichtigt.

2.2 Kernfunktionen im Überblick

  • Verwaltung von Mitarbeiterstammdaten (CRUD, CSV-Import/-Export)
  • Visuelle Absenzbuchung via Drag-and-Drop im Kalender
  • Automatische Berechnung verfügbarer Story Points pro Mitarbeiter, Team und Iteration
  • Dashboard mit KPI-Karten, Diagrammen und automatischer Lückenerkennung (Betrieb, Pikett)
  • PI Dashboard: Vergleich geplante SP (Jira) vs. berechnete SP (App)
  • Echtzeit-Synchronisation für mehrere gleichzeitige Benutzer via WebSocket
  • Mandantenfähigkeit: mehrere Agile Release Trains (ART) isoliert verwaltbar
  • Vollständige Export-Funktionen (PDF, PNG, JSON-Backup)
  • Corporate Design Bund (Bundesblau, Bundesrot, Swiss Design System)

2.3 Systemgrenzen (Out of Scope)

Folgende Bereiche sind explizit nicht Bestandteil dieser Ausschreibung:

  • Mobile App (nativer iOS/Android-Client)
  • SSO / Active Directory / LDAP Integration
  • SAP-Integration
  • Automatische Feiertagsberechnung (manuelle Pflege via CSV)
  • Jira REST API Integration (manuelle SP-Eingabe, keine automatische Synchronisation)
  • Custom Allocation Types (individuelle Buchungstypen – Roadmap)
  • Rollen und Berechtigungen (Admin / Planer / Read-Only – Roadmap)

3. Architektur-Übersicht

3.1 Systemarchitektur

Die Applikation folgt einer klassischen Fullstack-Architektur mit getrenntem Frontend und Backend. Beide Komponenten sind stateless und containerisierbar. Die Datenhaltung erfolgt server-seitig als JSON-basierter Dateispeicher (kein externes Datenbanksystem erforderlich).

Architekturprinzip: Der Server ist die einzige Source of Truth. Der Client hält keinen persistenten State — alle Daten werden beim Start via REST geladen und bei Änderungen via WebSocket synchronisiert.

Hinweis zu Filesystem-Persistenz: Die JSON-basierte Datenhaltung erfordert ein persistentes Filesystem auf dem Backend-Host. PaaS-Plattformen mit ephemerem Filesystem (z.B. 4. Architectural Decision Records (ADR)

Die folgenden ADRs dokumentieren die wesentlichen Architekturentscheide des Projekts mit Begründung und Konsequenzen. Sie sind verbindliche Orientierung für die Neuentwicklung.

ADR-001: JSON-Dateisystem statt relationale Datenbank
Entscheidung:Alle Anwendungsdaten werden als JSON-Dateien im Backend-Filesystem gespeichert. Kein SQL-ORM, keine externe Datenbank.
Begründung:Einfacheres Deployment (kein DB-Setup), kein Schema-Management, vollständige Daten-Portabilität via JSON-Backup. Für die Zielnutzerzahl (< 20 gleichzeitig) ausreichend performant.
Konsequenzen:Kein Locking bei konkurrierenden Schreibzugriffen (Last-Write-Wins, akzeptiert). Backup = vollständiges JSON-Snapshot. Persistentes Filesystem auf dem Host erforderlich.
Alternative:PostgreSQL oder SQLite möglich – Persistenzschicht ist intern hinter StateManager-Interface abstrahiert und kann ohne API-Änderung ausgetauscht werden.
Status:AKZEPTIERT
ADR-002: Server als einzige Source of Truth (Multiuser)
Entscheidung:Der Server hält den vollständigen AppData-State in Memory (mit File-Persistenz). Clients sind zustandslos und erhalten bei Verbindung den vollständigen State via REST.
Begründung:Konsistente Datensicht für alle gleichzeitigen Benutzer. Kein Merge-Konflikt-Management nötig. Einfache Reconnect-Logik (Client holt State neu).
Konsequenzen:Kein localStorage für Kerndaten. Socket.io für Echtzeit-Broadcast. Bei Server-Neustart: State wird aus JSON-Datei wiederhergestellt.
Verbote:LocalStorage darf ausschliesslich für UI-Präferenzen (z.B. Filter-Zustand) verwendet werden – nie für Geschäftsdaten.
Status:AKZEPTIERT
ADR-003: Blocker-Tage sind keine Absenzen (Geschäftsregel)
Entscheidung:Change-Freeze / Blocker-Tage zählen als normale Arbeitstage. Kein SP-Abzug. Nur visuelle Markierung im Kalender (Schneeflocken-Symbol ❄).
Begründung:Ein Freeze-Tag blockiert Deployments, nicht die Entwicklungskapazität. Mitarbeiter arbeiten weiterhin an Story Points.
Konsequenzen:SP-Berechnungsfunktion ignoriert Blocker-Einträge. Kalender-Renderer zeigt Schneeflocken-Overlay.
Status:AKZEPTIERT
ADR-004: Tenant-Isolation via Backend-Rooms und getrennte State-Dateien
Entscheidung:Jeder Agile Release Train (Mandant) erhält eine eigene State-Datei (state_{tenantId}.json) und einen eigenen Socket.io-Room. Daten verschiedener Trains sind auf Serverebene vollständig isoliert.
Begründung:BIT betreibt mehrere Trains (PS-NET, PS-INT etc.) die unabhängig planen müssen. Datenlecks zwischen Trains sind inakzeptabel.
Konsequenzen:TenantGate-Splash beim App-Start. Header zeigt aktiven Train. Socket.io-Events werden nur innerhalb des jeweiligen Rooms gebrodcastet.
Status:AKZEPTIERT
ADR-005: Pikett-Lücken gelten 7×24, Betrieb-Lücken nur an Arbeitstagen
Entscheidung:Die Lücken-Erkennung unterscheidet zwei Regelsets: Betrieb-Lücken werden nur an Werktagen (kein Wochenende, kein gesetzlicher Feiertag) gemeldet. Pikett-Lücken gelten jeden Tag, einschliesslich Wochenenden und Feiertagen.
Begründung:Betriebsdienste fallen an Weihnachten nicht an – keine Lücke. Pikett-Dienst (Störungsbereitschaft) gilt 365 Tage/Jahr.
Konsequenzen:Betrieb-Checker: isWorkday(date) && count < minBetrieb. Pikett-Checker: count < minPikett (kein Tages-Filter).
Status:AKZEPTIERT
ADR-006: Admin-Code mit bcrypt-Hash und server-seitigem Rate-Limiting
Entscheidung:Admin-Codes werden als bcryptjs-Hash in der Tenant-Registry gespeichert. Rate-Limiting: max. 3 Fehlversuche, dann 60 Sekunden Sperre (server-seitig, in-memory Map pro Tenant).
Begründung:Kein Plaintext-Passwort in der Konfiguration. Brute-Force-Schutz ohne externe Auth-Infrastruktur.
Einschränkungen:Kein echter 2FA. Code wird 15 Minuten im sessionStorage gecacht. Ausreichend für internen Pilotbetrieb.
Status:AKZEPTIERT TOTP als Roadmap-Feature R-01
ADR-007: Swiss Design System CSS-Variablen-Architektur (BIT Skin)
Entscheidung:Farbdefinitionen folgen dem Swiss Design System CSS-Custom-Properties-Schema (--color-primary-*, --color-secondary-*). BIT-Skin: Bundesblau als --color-primary-700, Bundesrot als --color-secondary-500.
Begründung:Zukunftssicherheit für Swiss-DS-Updates. Austauschbare Themes ohne Code-Änderungen. NotoSans als Frutiger-Fallback (Open Source, selbst-gehostet).
Konsequenzen:Keine externen Fonts via CDN. Alle Schriftarten lokal gehostet. Dynamische Hex-Farben (Buchungstypen) als inline styles (Tailwind JIT unterstützt keine Laufzeit-Hex-Werte).
Status:AKZEPTIERT
ADR-008: SP/Tag und Std/Jahr als globale Parameter (nicht pro Team)
Entscheidung:Story Points pro Tag und Arbeitsstunden pro Jahr sind systemweite Konstanten, konfigurierbar in den globalen Einstellungen. Individuelle Abweichungen werden über FTE-Faktor und Employee-spezifisches storyPointsPerDay abgebildet.
Begründung:Organisationsweite Standards sind einheitlich. Pro-Team-Konfiguration würde zu inkonsistenten Vergleichen zwischen Teams führen.
Status:AKZEPTIERT

interface BackupFile {
  schemaVersion:  string;    // z.B. "1.3" – IMMER erhöhen bei Model-Änderungen
  exportedAt:     string;    // ISO-8601 Zeitstempel
  tenantId?:      string;    // optional für Tenant-aware Backups

Kritische Regel: Die schemaVersion muss bei jeder Änderung am AppData-Interface erhöht werden. Alte Backups müssen via Migrations-Funktionen geladen werden können (Vorwärtskompatibilität). Backups ohne schemaVersion werden als Version 1.0 behandelt.

Alle Events sind auf den Tenant-Room tenant:{tenantId} beschränkt. Ein Client joined beim Verbindungsaufbau genau einen Room via tenant:join. Events anderer Trains sind nicht sichtbar.

Client → Server

EventPayloadBeschreibung

Server → Client (Broadcast an Room)

EventPayloadBeschreibung

allocation:change{ employeeId, dateStr, allocationType }Buchungsänderung – broadcast an alle Clients ausser Sender.
settings:change{ type, data }Settings-Änderung – broadcast an alle Clients ausser Sender.

Loop-Verhinderung

Eingehende Events (vom Server) dürfen keinen neuen ausgehenden Event auslösen. Client-seitig: Zustandsänderungen durch eingehende Events werden direkt via setState angewendet, ohne dass der useEffect-Hook einen neuen Emit auslöst. Empfohlenes Muster: isRemoteUpdate-Flag oder separate Setter für lokale vs. remote Änderungen.

10.2 Lückenerkennung (verbindliche Geschäftsregeln)

Betrieb-Lücke (gemäss ADR-005)

Eine Betrieb-Lücke wird gemeldet, wenn an einem Arbeitstag (kein Wochenende, kein gesetzlicher Feiertag) die Anzahl der Mitarbeiter mit Buchungstyp BETRIEB oder BETRIEB_PIKETT in einem Team kleiner ist als die konfigurierte Mindestbesetzung (teamConfig.minBetrieb).

Pikett-Lücke (gemäss ADR-005)

Eine Pikett-Lücke wird gemeldet, wenn an einem beliebigen Tag innerhalb des PI-Zeitraums (einschliesslich Wochenenden und gesetzlicher Feiertage) die Anzahl der Mitarbeiter mit Buchungstyp PIKETT oder BETRIEB_PIKETT in einem Team kleiner ist als die konfigurierte Mindestbesetzung (teamConfig.minPikett). Pikett gilt 7 × 24, da Störungen jederzeit auftreten können.

11.2 Sicherheitsanforderungen Neuentwicklung

  • Admin-Codes via bcrypt (Kostenfaktor ≥ 10) hashen.
  • Rate-Limiting ist server-seitig implementiert – nicht via Client-Logik aushebelbar.
  • Keine sensitiven Daten in URL-Parametern, Browser-History oder Logs.
  • CORS korrekt konfiguriert (nur eigene Domain als Origin).
  • Socket.io CORS-Origin muss zur Deployment-Domain passen (kein Wildcard in Produktion).
  • Tenant-IDs in Pfaden validieren (Regex-Check) – keine Path-Traversal-Angriffe möglich.

BegriffDefinition

Swiss DSSwiss Design System – offizielles Design System der Schweizer Bundesverwaltung
TenantGateSplash-Screen beim App-Start zur Auswahl des aktiven Trains
